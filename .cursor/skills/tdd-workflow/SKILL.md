# 测试驱动开发工作流

此 skill 通过 subagent 编排完成从需求澄清到代码审查的完整开发流程。

## !! 强制执行模式 !!

> **禁止主 agent 直接执行 TDD 步骤。必须通过 subagent 编排完成。**

主 agent 的角色是**编排者**，而不是执行者。唯一例外：需求澄清阶段由主 agent 直接与用户交互。

## 编排流程（四个阶段）

```
阶段 0  [主 agent]           需求澄清 → 消除歧义，确认范围
阶段 1  [planner subagent]   分析需求 → 输出 2-3 种方案
        [主 agent]           呈现方案 → 用户选择 ← 关键门控
阶段 2  [tdd-guide subagent] 按选定方案执行 TDD（可并行拆分）
阶段 3  [code-reviewer]      审查所有变更
```

---

## 阶段 0：需求澄清（主 agent 直接执行）

在启动任何 subagent 之前，主 agent **必须**先向用户澄清需求。

### 何时需要澄清

- 需求描述少于 2 句话
- 存在多种可能的理解方式
- 涉及技术选型决策（持久化、状态管理、UI 风格等）
- 功能边界不清晰（做什么 / 不做什么）

### 何时可以跳过

- 用户已提供详细的需求文档或设计稿
- 简单的工具函数或 bug 修复，意图明确
- 用户明确表示"不需要讨论，直接做"

### 澄清方式

使用 AskQuestion 工具或直接提问，收集以下信息：

1. **功能边界**：核心功能 vs 可选功能
2. **技术偏好**：是否有特定的实现方式偏好
3. **数据来源**：纯前端状态 / API 接口 / localStorage
4. **UI 参考**：是否有设计稿或参考页面
5. **优先级**：MVP 先做什么，后续再迭代什么

### 示例

```
用户: "新增一个 TodoList 页面"

主 agent 应该提问：
1. TodoList 需要哪些功能？（仅增删？还是需要编辑、分类、优先级、截止日期？）
2. 数据是否需要持久化？（纯内存 / localStorage / 后端 API）
3. 是否有 UI 设计稿？还是按项目现有风格自由发挥？
4. 是否需要拖拽排序等高级交互？
```

## 阶段 1：规划与方案选择

### 步骤 1：启动 planner subagent

使用 Task 工具，`subagent_type="planner"`：

```
prompt: |
  分析以下功能需求并提供 2-3 种实现方案：

  ## 用户需求
  [将用户需求和澄清后的结论完整传递]

  ## 要求
  1. 分析现有代码结构（路由、组件、i18n 等模式）
  2. 提供 2-3 种方案，每种方案包含：
     - 方案概述（1-2 句话）
     - 技术选型和架构决策
     - 需要创建/修改的文件清单
     - 复杂度估算（高/中/低）
     - 优点和缺点
  3. 给出推荐方案及理由
  4. 为推荐方案制定详细的实现计划和 TDD 测试策略
```

### 步骤 2：呈现方案，等待用户选择

**禁止在用户确认前启动 tdd-guide。**

主 agent 将 planner 的输出整理后呈现给用户：

```markdown
planner 分析了你的需求，提供了以下方案：

**方案 A：[名称]**（推荐）
- 概述：...
- 复杂度：中
- 优点：...
- 缺点：...

**方案 B：[名称]**
- 概述：...
- 复杂度：低
- 优点：...
- 缺点：...

你倾向于哪个方案？或者需要调整？
```

用户可以：
- 选择方案（"用方案 A"）
- 调整方案（"方案 A，但不需要 XX 功能"）
- 要求更多方案（"这两个都不太合适，能不能..."）

### 小型任务的简化流程

对于简单的工具函数、bug 修复等小型任务：
- planner 可以只输出 1 个方案（无需多方案）
- 主 agent 简要确认后即可进入实现阶段
- 判断标准：涉及 ≤3 个文件，无架构决策

---

## 阶段 2：TDD 实现

用户确认方案后，启动 tdd-guide subagent。

### 启动 tdd-guide subagent

使用 Task 工具，`subagent_type="tdd-guide"`：

```
prompt: |
  按照以下实现计划，使用 TDD 方法完成开发：
  [将用户选定的方案和详细计划传递给 tdd-guide]

  TDD 执行流程（每个模块重复）：
  1. RED   - 先编写失败的测试
  2. GREEN - 编写最小代码使测试通过
  3. REFACTOR - 重构优化
  4. 运行 pnpm test:run 验证
  5. 运行 pnpm test:coverage 验证覆盖率 80%+

  项目约束：
  - [传递关键的项目规则，如 i18n、命名规范等]
```

如果计划中有独立的模块（如 Store 和组件互不依赖），可以**并行启动多个 tdd-guide subagent**。

---

## 阶段 3：代码审查

### 启动 code-reviewer subagent

使用 Task 工具，`subagent_type="code-reviewer"`：

```
prompt: |
  审查以下 TDD 开发产出的代码变更：
  [列出所有变更的文件路径]

  审查要点：
  - 测试覆盖率是否达到 80%+
  - 代码是否符合项目规范
  - 是否存在安全问题
  - 是否有遗漏的边界情况
```

---

## 主 agent 的职责总结

| 阶段 | 主 agent 做什么 | 禁止做什么 |
|------|----------------|-----------|
| 阶段 0 | 直接向用户提问澄清需求 | - |
| 阶段 1 | 启动 planner，呈现方案，等待用户选择 | 禁止跳过用户确认 |
| 阶段 2 | 启动 tdd-guide subagent | 禁止直接编写代码/测试 |
| 阶段 3 | 启动 code-reviewer subagent | 禁止跳过审查 |
| 最后 | 汇总结果，回复用户 | - |

---

## 传递给 subagent 的 TDD 核心原则

以下内容应作为上下文传递给 `tdd-guide` subagent：

### 覆盖率要求

- 最低 80% 覆盖率（单元 + 组件 + E2E）
- 覆盖所有边界情况和错误场景

### 测试类型

| 类型 | 覆盖范围 |
| --- | --- |
| 单元测试 | 工具函数、Composables、纯函数、数据处理逻辑 |
| 组件测试 | 渲染、Props/Emits、交互、插槽、状态变化 |
| E2E 测试 | 关键用户流程、完整工作流、UI 交互 |

### TDD 循环

```
RED    → 编写失败的测试
GREEN  → 编写最小代码使测试通过
REFACTOR → 重构代码，保持测试通过
```

### 测试命令

```bash
pnpm test:run       # 单次运行测试
pnpm test:coverage  # 运行测试并生成覆盖率报告
pnpm test           # 监听模式
```

## 成功指标

- 需求已与用户确认，无歧义
- 用户已选择实现方案
- 达到 80%+ 代码覆盖率
- 所有测试通过
- 所有代码通过 code-reviewer 审查
