# 提交规范参考

## 提交最佳实践

- **提交前验证**：确保代码已检查且文档已更新
- **原子提交**：每个提交应包含服务于单一目的的相关更改
- **拆分大型更改**：如果更改涉及多个关注点，将其拆分为单独的提交
- **约定式提交格式**：使用 `<类型>: <描述>` 格式，类型包括：
  - `feat`：新功能
  - `fix`：错误修复
  - `docs`：文档更改
  - `style`：代码风格更改（格式化等）
  - `refactor`：既不修复错误也不添加功能的代码更改
  - `perf`：性能改进
  - `test`：添加或修复测试
  - `chore`：构建过程、工具等的更改
- **现在时态，祈使语气**：将提交信息写成命令（如"添加功能"而非"添加了功能"）
- **简洁的首行**：首行保持在 72 个字符以内

## 拆分提交的准则

分析 diff 时，根据以下标准考虑拆分提交：

1. **不同关注点**：对代码库不相关部分的更改
2. **不同类型的更改**：混合功能、修复、重构等
3. **文件模式**：对不同类型文件的更改（如源代码 vs 文档）
4. **逻辑分组**：单独理解或审查会更清晰的更改
5. **大小**：如果分解会更清晰的非常大的更改

## 示例

好的提交信息：

- ✨ feat: 添加用户认证系统
- 🐛 fix: 解决渲染过程中的内存泄漏
- 📝 docs: 更新 API 文档添加新端点
- ♻️ refactor: 简化解析器中的错误处理逻辑
- 🚨 fix: 解决组件文件中的检查器警告
- 🧑‍💻 chore: 改善开发者工具设置流程
- 👔 feat: 实现交易验证的业务逻辑
- 🩹 fix: 修复头部中的小样式不一致
- 🚑️ fix: 修补认证流程中的关键安全漏洞
- 🎨 style: 重组组件结构以提高可读性
- 🔥 fix: 移除已弃用的遗留代码
- 🦺 feat: 为用户注册表单添加输入验证
- 💚 fix: 解决失败的 CI 管道测试
- 📈 feat: 实现用户参与度的分析跟踪
- 🔒️ fix: 加强认证密码要求
- ♿️ feat: 改善表单对屏幕阅读器的可访问性

拆分提交的示例：

- 第一个提交：✨ feat: 添加新的 solc 版本类型定义
- 第二个提交：📝 docs: 更新新 solc 版本的文档
- 第三个提交：🔧 chore: 更新 package.json 依赖
- 第四个提交：🏷️ feat: 为新 API 端点添加类型定义
- 第五个提交：🧵 feat: 改善工作线程中的并发处理
- 第六个提交：🚨 fix: 解决新代码中的检查问题
- 第七个提交：✅ test: 为新 solc 版本功能添加单元测试
- 第八个提交：🔒️ fix: 更新存在安全漏洞的依赖
